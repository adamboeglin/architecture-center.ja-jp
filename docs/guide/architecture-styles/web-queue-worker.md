# <a name="web-queue-worker-architecture-style"></a><span data-ttu-id="32313-103">Web キュー ワーカーのアーキテクチャ スタイル</span><span class="sxs-lookup"><span data-stu-id="32313-103">Web-Queue-Worker architecture style</span></span>

<span data-ttu-id="32313-104">このアーキテクチャのコア コンポーネントは、クライアント要求を処理する **Web フロント エンド**と、リソースを消費するタスク、実行時間の長いワークフロー、またはバッチ ジョブを実行する**ワーカー**です。</span><span class="sxs-lookup"><span data-stu-id="32313-104">The core components of this architecture are a **web front end** that serves client requests, and a **worker** that performs resource-intensive tasks, long-running workflows, or batch jobs.</span></span>  <span data-ttu-id="32313-105">Web フロント エンドは**メッセージ キュー**を通じてワーカーと通信します。</span><span class="sxs-lookup"><span data-stu-id="32313-105">The web front end communicates with the worker through a **message queue**.</span></span>

![Web キュー ワーカーのアーキテクチャ スタイルの論理図](./images/web-queue-worker-logical.svg)

<span data-ttu-id="32313-107">このアーキテクチャで一般的に使用されているその他のコンポーネントは次のとおりです。</span><span class="sxs-lookup"><span data-stu-id="32313-107">Other components that are commonly incorporated into this architecture include:</span></span>

- <span data-ttu-id="32313-108">1 つまたは複数のデータベース。</span><span class="sxs-lookup"><span data-stu-id="32313-108">One or more databases.</span></span>
- <span data-ttu-id="32313-109">データベースから高速でデータを読み取るために値を格納するキャッシュ。</span><span class="sxs-lookup"><span data-stu-id="32313-109">A cache to store values from the database for quick reads.</span></span>
- <span data-ttu-id="32313-110">静的コンテンツを提供するための CDN</span><span class="sxs-lookup"><span data-stu-id="32313-110">CDN to serve static content</span></span>
- <span data-ttu-id="32313-111">電子メールや SMS サービスなどのリモート サービス。</span><span class="sxs-lookup"><span data-stu-id="32313-111">Remote services, such as email or SMS service.</span></span> <span data-ttu-id="32313-112">多くの場合、これらは、サード パーティによって提供されます。</span><span class="sxs-lookup"><span data-stu-id="32313-112">Often these are provided by third parties.</span></span>
- <span data-ttu-id="32313-113">認証 ID プロバイダー。</span><span class="sxs-lookup"><span data-stu-id="32313-113">Identity provider for authentication.</span></span>

<span data-ttu-id="32313-114">Web とワーカーはいずれもステートレスです。</span><span class="sxs-lookup"><span data-stu-id="32313-114">The web and worker are both stateless.</span></span> <span data-ttu-id="32313-115">セッション状態は、分散キャッシュに格納することができます。</span><span class="sxs-lookup"><span data-stu-id="32313-115">Session state can be stored in a distributed cache.</span></span> <span data-ttu-id="32313-116">ワーカーによる実行時間の長い作業は、非同期的に実行されます。</span><span class="sxs-lookup"><span data-stu-id="32313-116">Any long-running work is done asynchronously by the worker.</span></span> <span data-ttu-id="32313-117">ワーカーは、キューのメッセージによってトリガーされるか、またはバッチ処理のためにスケジュールに従って実行できます。</span><span class="sxs-lookup"><span data-stu-id="32313-117">The worker can be triggered by messages on the queue, or run on a schedule for batch processing.</span></span> <span data-ttu-id="32313-118">ワーカーは、オプションのコンポーネントです。</span><span class="sxs-lookup"><span data-stu-id="32313-118">The worker is an optional component.</span></span> <span data-ttu-id="32313-119">処理時間が長い操作がない場合は、ワーカーを省略できます。</span><span class="sxs-lookup"><span data-stu-id="32313-119">If there are no long-running operations, the worker can be omitted.</span></span>

<span data-ttu-id="32313-120">フロント エンドは、Web API で構成されている可能性があります。</span><span class="sxs-lookup"><span data-stu-id="32313-120">The front end might consist of a web API.</span></span> <span data-ttu-id="32313-121">クライアント側では、Web API は、AJAX 呼び出しを実行する単一ページ アプリケーションまたはネイティブ クライアント アプリケーションが使用できます。</span><span class="sxs-lookup"><span data-stu-id="32313-121">On the client side, the web API can be consumed by a single-page application that makes AJAX calls, or by a native client application.</span></span>

## <a name="when-to-use-this-architecture"></a><span data-ttu-id="32313-122">このアーキテクチャを使用する状況</span><span class="sxs-lookup"><span data-stu-id="32313-122">When to use this architecture</span></span>

<span data-ttu-id="32313-123">Web キュー ワーカーのアーキテクチャは、通常、マネージド コンピューティング サービス、Azure App Service または Azure Cloud Services のいずれかを使用して実装されます。</span><span class="sxs-lookup"><span data-stu-id="32313-123">The Web-Queue-Worker architecture is typically implemented using managed compute services, either Azure App Service or Azure Cloud Services.</span></span>

<span data-ttu-id="32313-124">次の場合に、このアーキテクチャ スタイルを検討してください。</span><span class="sxs-lookup"><span data-stu-id="32313-124">Consider this architecture style for:</span></span>

- <span data-ttu-id="32313-125">比較的単純なドメインのアプリケーション。</span><span class="sxs-lookup"><span data-stu-id="32313-125">Applications with a relatively simple domain.</span></span>
- <span data-ttu-id="32313-126">実行時間の長いワークフローやバッチ操作があるアプリケーション。</span><span class="sxs-lookup"><span data-stu-id="32313-126">Applications with some long-running workflows or batch operations.</span></span>
- <span data-ttu-id="32313-127">サービスとしてのインフラストラクチャ (IaaS) ではなく、マネージド サービスを使用する場合。</span><span class="sxs-lookup"><span data-stu-id="32313-127">When you want to use managed services, rather than infrastructure as a service (IaaS).</span></span>

## <a name="benefits"></a><span data-ttu-id="32313-128">メリット</span><span class="sxs-lookup"><span data-stu-id="32313-128">Benefits</span></span>

- <span data-ttu-id="32313-129">比較的単純でわかりやすいアーキテクチャ。</span><span class="sxs-lookup"><span data-stu-id="32313-129">Relatively simple architecture that is easy to understand.</span></span>
- <span data-ttu-id="32313-130">展開と管理が容易。</span><span class="sxs-lookup"><span data-stu-id="32313-130">Easy to deploy and manage.</span></span>
- <span data-ttu-id="32313-131">懸念事項の明確な分離。</span><span class="sxs-lookup"><span data-stu-id="32313-131">Clear separation of concerns.</span></span>
- <span data-ttu-id="32313-132">フロント エンドは、非同期メッセージングを使用してワーカーから切り離されます。</span><span class="sxs-lookup"><span data-stu-id="32313-132">The front end is decoupled from the worker using asynchronous messaging.</span></span>
- <span data-ttu-id="32313-133">フロント エンド ロールとワーカーはそれぞれ個別に拡張できます。</span><span class="sxs-lookup"><span data-stu-id="32313-133">The front end and the worker can be scaled independently.</span></span>

## <a name="challenges"></a><span data-ttu-id="32313-134">課題</span><span class="sxs-lookup"><span data-stu-id="32313-134">Challenges</span></span>

- <span data-ttu-id="32313-135">慎重に設計しないと、フロント エンドおよびワーカーが、大型のモノリシック コンポーネントになり、管理や更新が困難になる。</span><span class="sxs-lookup"><span data-stu-id="32313-135">Without careful design, the front end and the worker can become large, monolithic components that are difficult to maintain and update.</span></span>
- <span data-ttu-id="32313-136">フロント エンド ロールとワーカーがデータ スキーマまたはコード モジュールを共有している場合、依存関係が潜在することがある。</span><span class="sxs-lookup"><span data-stu-id="32313-136">There may be hidden dependencies, if the front end and worker share data schemas or code modules.</span></span>

## <a name="web-queue-worker-on-azure-app-service"></a><span data-ttu-id="32313-150">Azure App Service の Web キュー ワーカー</span><span class="sxs-lookup"><span data-stu-id="32313-150">Web-Queue-Worker on Azure App Service</span></span>

<span data-ttu-id="32313-151">このセクションでは、Azure App Service を使用する、推奨 Web キュー ワーカー アーキテクチャについて説明します。</span><span class="sxs-lookup"><span data-stu-id="32313-151">This section describes a recommended Web-Queue-Worker architecture that uses Azure App Service.</span></span>

![Web キュー ワーカーのアーキテクチャ スタイルの物理図](./images/web-queue-worker-physical.png)

- <span data-ttu-id="32313-153">フロント エンドは、Azure Functions アプリとして実装され、ワーカーは、WebJob として実装されます。</span><span class="sxs-lookup"><span data-stu-id="32313-153">The front end is implemented as an Azure App Service web app, and the worker is implemented as an Azure Functions app.</span></span> <span data-ttu-id="32313-154">Web アプリと関数アプリの両方が VM インスタンスを提供する App Service プランに関連付けられています。</span><span class="sxs-lookup"><span data-stu-id="32313-154">The web app and the function app are both associated with an App Service plan that provides the VM instances.</span></span>

- <span data-ttu-id="32313-155">メッセージ キューには、Azure Service Bus または Azure Storage キューのいずれかを使用できます。</span><span class="sxs-lookup"><span data-stu-id="32313-155">You can use either Azure Service Bus or Azure Storage queues for the message queue.</span></span> <span data-ttu-id="32313-156">(図は、Azure Storage キューを示しています)。</span><span class="sxs-lookup"><span data-stu-id="32313-156">(The diagram shows an Azure Storage queue.)</span></span>

- <span data-ttu-id="32313-157">Azure Redis Cache は、セッション状態と短い待ち時間でアクセスが必要なその他のデータを格納します。</span><span class="sxs-lookup"><span data-stu-id="32313-157">Azure Redis Cache stores session state and other data that needs low latency access.</span></span>

- <span data-ttu-id="32313-158">Azure CDN を使用して、画像、CSS、HTML などの静的なコンテンツをキャッシュします。</span><span class="sxs-lookup"><span data-stu-id="32313-158">Azure CDN is used to cache static content such as images, CSS, or HTML.</span></span>

- <span data-ttu-id="32313-159">ストレージは、アプリケーションのニーズに最適なストレージ テクノロジを選択します。</span><span class="sxs-lookup"><span data-stu-id="32313-159">For storage, choose the storage technologies that best fit the needs of the application.</span></span> <span data-ttu-id="32313-160">複数のストレージ テクノロジ (多言語持続性) を使用することもできます。</span><span class="sxs-lookup"><span data-stu-id="32313-160">You might use multiple storage technologies (polyglot persistence).</span></span> <span data-ttu-id="32313-161">この概念を示すために、この図には、Azure SQL Database と Azure Cosmos DB を示しています。</span><span class="sxs-lookup"><span data-stu-id="32313-161">To illustrate this idea, the diagram shows Azure SQL Database and Azure Cosmos DB.</span></span>

<span data-ttu-id="32313-162">詳細については、[App Service Web アプリケーションの参照アーキテクチャ][scalable-web-app]を参照してください。</span><span class="sxs-lookup"><span data-stu-id="32313-162">For more details, see [App Service web application reference architecture][scalable-web-app].</span></span>

### <a name="additional-considerations"></a><span data-ttu-id="32313-163">追加の考慮事項</span><span class="sxs-lookup"><span data-stu-id="32313-163">Additional considerations</span></span>

- <span data-ttu-id="32313-164">ストレージまでキューやワーカーを経由しないトランザクションもあります。</span><span class="sxs-lookup"><span data-stu-id="32313-164">Not every transaction has to go through the queue and worker to storage.</span></span> <span data-ttu-id="32313-165">Web フロント エンドは、単純な読み取り/書き込み操作を直接実行できます。</span><span class="sxs-lookup"><span data-stu-id="32313-165">The web front end can perform simple read/write operations directly.</span></span> <span data-ttu-id="32313-166">ワーカーは、リソースを消費するタスクまたは実行時間の長いワークフローの設計されています。</span><span class="sxs-lookup"><span data-stu-id="32313-166">Workers are designed for resource-intensive tasks or long-running workflows.</span></span> <span data-ttu-id="32313-167">ワーカーがまったく必要ない場合もあります。</span><span class="sxs-lookup"><span data-stu-id="32313-167">In some cases, you might not need a worker at all.</span></span>

- <span data-ttu-id="32313-168">App Service の組み込み自動スケール機能を使用して、VM インスタンスの数をスケール アウト。</span><span class="sxs-lookup"><span data-stu-id="32313-168">Use the built-in autoscale feature of App Service to scale out the number of VM instances.</span></span> <span data-ttu-id="32313-169">アプリケーションで負荷が、予測可能なパターン通りの場合は、スケジュールに基づく自動スケールを使用します。</span><span class="sxs-lookup"><span data-stu-id="32313-169">If the load on the application follows predictable patterns, use schedule-based autoscale.</span></span> <span data-ttu-id="32313-170">負荷が予測可能でない場合は、メトリックに基づいた自動スケール ルールを使用します。</span><span class="sxs-lookup"><span data-stu-id="32313-170">If the load is unpredictable, use metrics-based autoscaling rules.</span></span>

- <span data-ttu-id="32313-171">Web アプリと関数アプリを個別の App Service プランに配置することを検討してください。</span><span class="sxs-lookup"><span data-stu-id="32313-171">Consider putting the web app and the function app into separate App Service plans.</span></span> <span data-ttu-id="32313-172">これにより、個別にスケールできます。</span><span class="sxs-lookup"><span data-stu-id="32313-172">That way, they can be scaled independently.</span></span>

- <span data-ttu-id="32313-175">デプロイ スロットを使用して展開を管理します。</span><span class="sxs-lookup"><span data-stu-id="32313-175">Use deployment slots to manage deployments.</span></span> <span data-ttu-id="32313-176">これにより、ステージング スロットに更新されたバージョンを展開してた後、新しいバージョンに交換できます。</span><span class="sxs-lookup"><span data-stu-id="32313-176">This lets you to deploy an updated version to a staging slot, then swap over to the new version.</span></span> <span data-ttu-id="32313-177">また、更新バージョンで問題があった場合は、前のバージョンに戻ることもできます。</span><span class="sxs-lookup"><span data-stu-id="32313-177">It also lets you swap back to the previous version, if there was a problem with the update.</span></span>
